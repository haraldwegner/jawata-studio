# jawata-studio v3.1.0 — the conductor delivery

The Sprint-25 seats reach the front-door agents. The conductor is not a new
program: it IS the agent you already talk to (Claude Code, Cursor's agent,
Antigravity's agents) — this release gives that agent the seats through
jawata's existing delivery model. Sprint 25a; jawata-mcp is unchanged
(toolCount stays 45, no mcp release).

## Generated seat commands, per client (D1)

From each seat file (`seats/*.md` — the single source, shipped in the binary
and materialized into the studio config dir, where a user edit survives every
redeploy and regenerates every channel), the deploy now writes a per-client
command with the seat's stance embedded and the Lane-1 loop contract on top
(detect → do → verify with NAMED jawata gate calls → PROPOSE, never
auto-apply → record the outcome to the experience store):

| Command | Seat | Channel per client |
|---|---|---|
| `/javadocs` | javadoc-writer | Claude Code `.claude/skills/` · Cursor `commands/` · Antigravity `.agent/workflows/` |
| `/cover` | test-writer | (same three channels) |
| `/refactor` | architect | (same) |
| `/debug` | debugger | (same) |
| `/profile` | profiler | (same) |

Claude Desktop / claude.ai: the deploy exports ONE `jawata-seats` skill zip
(vendor format: a single skill folder at the zip root; the five seats ride as
`references/<command>.md`) into the studio export dir — upload it once; the
deploy message names the path. IntelliJ has no command channel; the rule
block's phrase table substitutes. The spec-editor and spec-auditor seats live
in the `/sprint` pipeline, not as commands.

Everything is generated — none of these files is hand-authored — and the
deploy is idempotent at the file level: a byte-identical redeploy writes
nothing, Regenerate force-rewrites, Delete removes exactly the generated
artifacts and prunes the directories it created.

## The conductor section in the managed rule block (D2)

Every managed client's rule block now teaches the front-door agent: the seat
catalog, when to involve the architect unprompted (vague "clean this up"
asks, checkpoint diffs), and the seat discipline — gates are real jawata
calls, a gate you could not run has NOT passed, propose-mode, every outcome
recorded to the store. Per-client tail: a one-liner where commands are
installed; the full phrase table on IntelliJ. The section is budget-guarded
(a build-failing line limit) and is the one deliberate per-client variation
in the otherwise identical block.

## Design mode enters the /sprint pipeline (D3)

The architect's design mode (v3.0.0) is now a mandatory pipeline step: after
spec sign-off, before the plan, a design-mode run produces the versioned
`ARCHITECTURE-<scope>.md` (with picture); plan promotion requires it. This
release's own generator subsystem was designed exactly this way
(`ARCHITECTURE-conductor.md`).

## Honest limits (recorded decisions)

- **Lane 1 is taught, not enforced** (R3, accepted): a front-door agent
  running a seat by stance has no hard ceilings or purity invariant — those
  are the hosted runner's (Lane B, Sprint 28); hook-level seat enforcement is
  a named Sprint-26 deliverable. The gates themselves are real jawata calls
  that refuse dishonestly-green input, and skipped gates are visible in the
  store.
- **Antigravity + claude.ai are first deployments** (R1): both channel
  formats were verified against current vendor docs at implementation
  (`.agent/workflows/`, one-skill-per-zip); if a live install still differs,
  the phrase-table fallback is the shipped state for that client, recorded.
- The rule block grows by the conductor section in every session (R2,
  accepted universally, budget-capped).
