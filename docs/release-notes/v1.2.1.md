# goja-studio v1.2.1

A dogfood-driven tuning of the v1.2.0 **PreToolUse enforcement hook**. No new
features — it narrows the guard so it stops over-blocking legitimate commands.

## Fix — the Java-search guard no longer over-matches

v1.2.0's hook redirected *any* `Bash` search command whose text merely mentioned
`.java`. Live dogfooding surfaced false positives: `find . -name "*.java"` (listing
files), `sed -n '1,20p' Foo.java` (reading a range), and grepping a build file or
log for the literal string `.java` were all wrongly blocked.

v1.2.1 requires **both**:

1. **A content-search tool** — `grep`/`egrep`/`fgrep`/`rg`/`ag`/`ack` (or the Grep
   tool). File/line ops (`find`, `sed`, `awk`) are no longer treated as symbol
   search and pass untouched.
2. **A real `.java` file target** — a concrete path (`Foo.java`, `src/Foo.java`) or
   a glob (`*.java`). An escaped regex pattern like `\.java` or an incidental
   mention of `.java` in a build file / log no longer trips the guard.

The genuine case — `grep`/`rg` for a symbol inside `.java` files — is still
redirected to GOJA's compiler-accurate tools, health-gated as before.

Verified against a live payload matrix: real symbol searches (grep in `Main.java`,
Grep glob `*.java`, `--include=*.java`, `rg -g *.java`) still block; the four
false-positive classes above now pass; non-Java commands are unchanged.

## Install

Download the installer for your platform from the assets below. Re-run the Linux
one-liner or reinstall the package to update; the guard is redeployed to
`~/.claude/goja-studio/pretooluse-guard.sh` on the next deploy.
