# jawata-studio v3.2.1 — release-pointer fix (Sprint 26 dogfood)

A dogfood patch. It carries the same client-side surface as v3.2.0 (the Stop
gate, the observer's consequence-labeled edit feed, and the `/memorize` +
`/train` utility commands) and fixes the release plumbing that stopped v3.2.0
from reaching installers.

## The install-pointer bug (finding #6)

`install.sh` (and `jawata.org/install.sh`) resolve GitHub's
`/releases/latest` to pick the app to download. The release workflow created
each release as a draft and left the "make this the latest release" step to a
manual step that missed it — so after v3.2.0 published, the studio
`/releases/latest` pointer stayed on **v3.1.0**, and every fresh install
received the old app. A v3.1.0 app deploys v3.1.0 hooks: no Stop gate, no edit
feed. The engine (jawata-mcp) was unaffected — it is fetched by a separate
path that resolved correctly.

## The fix

The release workflow now ends with a single `publish` job that runs after all
platform bundles have uploaded to the draft. It publishes the release, sets it
as `latest`, pins the release notes from `docs/release-notes/<tag>.md`, and
**fails the release loudly if `/releases/latest` does not then return the new
tag**. The per-platform matrix still uploads to a draft first, so a release is
never live while it is still missing an architecture's asset.

No application-code changes.
