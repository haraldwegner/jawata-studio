# goja-studio v1.5.1 — Authoring window (edit-gate refinement)

The v1.4.0 edit-gate blocks a hand-edit of a `.java` file to push structural changes through GOJA's
refactor tools. But **authoring new code is not a refactor** — `rename_symbol` / `extract` / `move`
can't express "add this new field/method" — and a text-level hook **cannot reliably tell authoring
from restructuring** (that judgment needs the AST; it is the intelligent-injector's job). So the gate
over-blocked legitimate new-code authoring, and the `goja-fallback:` escape had no clean channel for a
structured Edit/Write (its only text fields write to the file).

## The authoring window

Declare an authoring window with a Bash command containing **`goja-author: <reason>`**. It opens a
**session-scoped, TTL-bounded (30 min)** window during which `.java` edits pass — with **no marker
polluting the source**. The declaration and each covered edit are appended to the versioned
`~/.claude/goja-studio/fallback.log`, so the audit trail (and the training data for the
intelligent-injector) stays complete.

- The block still fires by default (first edit → "USE A GOJA REFACTOR TOOL", now also pointing at the
  authoring window).
- The window is a **declaration, not a classification** — the guard does not guess structural-vs-trivial
  (which needs the AST); the agent declares intent and it is logged. This is the deliberate cold-start
  rule; the AST-grounded structural-edit classifier (the intelligent-injector sprint) is its principled
  successor, trained on exactly these declarations.

## PostToolUse observer — false-slip fix

The v1.5.0 observer logged a "slip" whenever *any* tool input contained the string `goja-fallback:` —
including an edit whose **content** merely mentions the marker (e.g. editing the guard source itself),
polluting `outcomes.log` with false slips. This is the POST analog of the PRE v1.2.0 over-match.

v1.5.1 scopes slip-detection to a **real `.java`-gated op** (mirroring the PRE guard's dual-gate): an
`Edit`/`Write`/`MultiEdit` whose `file_path` is `.java`, or a `grep`-on-`.java` Bash command — plus the
marker. A non-`.java` edit or a non-search Bash command that incidentally contains the marker is no
longer counted. The scoring substrate stays clean.

## Compatibility

Additive to the PreToolUse guard + PostToolUse observer; escapable, session-scoped, TTL-bounded;
non-Java work untouched. No change to any tool.
