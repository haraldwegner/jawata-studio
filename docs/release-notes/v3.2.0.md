# jawata-studio v3.2.0 — the injector's client side (Sprint 26)

Companion to jawata-mcp v3.2.0 (the intelligent injector). The studio's half:
the hooks that enforce and feed the learning loop in Claude Code, and the
generated utility commands that give every client a memory and a training
door.

## The Stop gate (D5 bounce + D4 block, Claude Code)

A managed Stop hook (deployed and removed like the guard/observer quartet)
reads the session's final message and gates the three summary shapes —
decision ask, checkpoint summary, sprint result:

- Applies the mechanical DECISION TEST: decision named early, options
  present, no undefined ALL-CAPS terms, length cap. Fail → the message is
  bounced back with the reason, ONE rewrite loop (never an infinite bounce),
  and the bounce is recorded to the experience store.
- Seat-gate block: a session that loaded a seat command and reaches its
  final proposal with NO gate call in the transcript is blocked with the
  missing gates named.
- Fail-open everywhere: an unreadable transcript, missing python, or a dead
  resident never traps the user.

Cursor has no stop-equivalent hook event (platform fact, recorded at plan
promotion); Cursor is served by the resident's server-side lane instead.

## The observer's consequence-labeled edit feed (C7)

The PostToolUse observer now feeds the resident's edit switch from real
edits with their REAL outcome: a `.java` edit's before/after fragments are
held per client session; the session's next compile/diagnostics/tests
outcome — or an undo, which forces "failed" — labels them and posts each as
`experience(kind=observe_edit, outcome=clean|failed)`. Pop-first semantics:
a lost post is a lost label, never a stale re-label; an unreadable outcome
means no label, never a guessed one. Fail-open: no python3 or resident down
= no feed, everything else unaffected.

## /memorize + /train — generated utility commands (D6/D7)

Two utility commands join the five seat commands on the managed deploy
lifecycle (written, verified, removed per client — Claude Code skills,
Cursor commands, Antigravity workflows, and the claude.ai skill zip, now 8
entries):

- **/memorize** — store-first: the durable fact goes to the experience
  store (cross-client) FIRST, then file memory; bare form stores what is
  durable from the discussion, argument form stores exactly that.
- **/train** — calls `experience(kind=train)` / `(kind=learner_status)` and
  reports the learners' honest state (events by kind, DECIDES/SHADOW with
  rolling records).

## The verdict bridge (D7)

`record_human_verdict` no longer stops at the journal file: the
verdict-enriched seat-run entry is re-recorded to the experience store — the
human-labeled corpus row the seat learners train on. A store-sync failure is
LOUD in the result, never swallowed.
