# jawata-studio v3.4.1 — carries the mcp fix, and a sharper sprint rule

This release exists to carry **jawata-mcp v3.4.1**, which repairs a defect in
v3.4.0: the store's meaning-based recall was present, tested, and never actually
run by the product. Asking a question in different words got the old keyword
search. See the mcp release notes for the full account.

## What changed here

Nothing in how the studio behaves. Version aligned to 3.4.1 so the pair ships
together.

The one substantive change is to the `/sprint` process the studio deploys to
every client. Three additions, all from the v3.4.0 failure:

**An end-to-end test runs before the release sign-off ask, not after.** Every
deliverable is exercised through the product's own front door, against the built
artifact, and the results go into the ask. A deliverable with no such check is
reported as unproven rather than done. Putting a check like this only in release
CI does not count — CI runs after the human has already approved.

**"Called only by tests" is a smell, and it is checked.** A public member whose
callers are all in test code is the mechanical signature of a capability that was
built and never connected. It would have named both halves of the v3.4.0 defect
before release.

**The architect reviews the complete deliverable set at the end of the sprint**,
naming the production code that calls each capability — file and line, in a main
source root. The per-checkpoint review looks at each stage's changes, and a
missing connection is a *non-edit*: it lives in a line no commit touched, appears
in no diff, and breaks no assertion. Sprint 27's own missing connection sat in a
line last modified twelve days before that sprint began.

Why all three, rather than one: the automated tests were green, four different
ways, and the coverage measurement rose, while the feature did nothing. None of
those instruments can see a capability that is never called, because each test
supplies the connection itself.
