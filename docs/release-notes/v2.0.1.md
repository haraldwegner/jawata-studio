# goja-studio v2.0.1 — push-hook peel fix (dogfood finding)

A live dogfood of the v2.0.0 build over its real HTTP transport caught a bug the unit tests
couldn't: every successful MCP result carries a `,"meta":{"steering":…}` block **after** `"data"`
(the Sprint-22 result-steering layer). The push hooks' `data`-extraction used a greedy
`"\(.*\)"` sed, which **swallowed the trailing meta blob** into the injected `additionalContext`.

## The fix

Both push scripts (SessionStart primer + PreToolUse recall) now extract `data` with `"\([^"]*\)"`
— stopping at the data string's closing quote instead of the last quote on the line. This is safe
**because the mcp render sanitizes quotes out of the data lines** (v2.0.0), so the value is
quote-free and `[^"]*` captures exactly it. Verified against the real double-encoded, meta-carrying
response from the live v2.0 resident.

A regression test (`push_scripts_extract_data_without_swallowing_meta`) guards both templates
against a return to the greedy form. `cargo test`: 63 passed.

No mcp change — goja-mcp v2.0.0 is unaffected (the bug was purely in the studio-side bash peel).
