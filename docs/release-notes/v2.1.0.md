# goja-studio v2.1.0 — Knowledge view, auto-seed, centralized backups, push-channel fixes (Sprint 21a)

The studio half of Sprint 21a, pairing with **goja-mcp v2.1.0**.

## The Knowledge / Database view

A new top-level **Knowledge** tab — the store made visible and operable. Its actions carry
**exactly the prompt vocabulary**: `load · reseed · wipe · refresh · list · promote ·
export · import · prune · dedup · compact` — what you click is what you'd say to the agent,
and every button shows its prompt phrase as a hint.

- **Store overview** per workspace: entry counts by status/language, store file + size,
  reachability.
- **wipe / reseed** with explicit confirmation (reseed = the atomic wipe + reload from your
  configured memory roots).
- **Memory sources:** root folders editor, recursive toggle, crawl caps (depth/files/bytes),
  and the **store mode** — `shared` (one user-level store across all workspaces — the
  default) / `workspace` / `memory`. Everything reaches the resident as JVM properties.
- **Curation:** list candidates (and rejected/superseded), promote/reject inline.
- **Backups:** retention setting + a one-time **GC** for historically scattered `.bak`
  files (dry-run report first; unrecognized files are never touched).

## Auto-seed on deploy

After a successful deploy the studio calls `experience(kind=load)` on every resident
(background, fail-safe, toggleable — default on), so the knowledge push channel has content
from day one instead of answering with absence until someone seeds by hand.

## Centralized backups — no more `.bak` litter

Every managed write used to drop `<file>.bak-<epoch-ms>` NEXT TO your files (`~/.claude`,
project dirs, client configs) with no retention. All backups now land in ONE managed area
(`<data_root>/backups/…`), keep the last N versions (default 10, configurable), and the
Knowledge view's GC sweeps the old scattered files in.

## Push-channel correctness

- The PostToolUse observer judges the **request only** — previously a `cat` of a file whose
  *content* mentioned `.java` + `goja-fallback:` logged a false slip.
- **Post-deploy hook self-check:** after writing each hook, the deploy drives its selftest
  path (which shares the live emit format) and validates the **emitted bytes** are valid
  JSON — the v2.0.1/v2.0.2 bug class now fails the deploy instead of shipping.
- **Slips auto-learn:** a declared `goja-fallback:` slip is also recorded into the
  experience store as a `failure_mode` candidate (fail-safe when the resident is down).

## Notes

- Deploy once after updating so the corrected hook templates replace the older copies on
  disk.
- Existing settings are preserved; the new knowledge/backup preferences default sensibly
  (shared store, auto-seed on, retention 10).
