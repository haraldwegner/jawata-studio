# jawata-studio v3.2.2 — Deploy-to-Claude fix (Sprint 26 dogfood)

A dogfood patch. It fixes the broken Claude deploy introduced with the v3.2.0
Stop gate.

## The broken deploy (finding #7)

Deploying to Claude failed with:

> claude: hook self-check output lacks hookSpecificOutput.additionalContext:
> …/stop-gate.sh

The post-deploy self-check validates a hook by running its selftest path and
checking the emitted JSON against the hook's contract. It required
`hookSpecificOutput.additionalContext` — the contract for context-INJECTING
hooks (the observer, the recall/primer hooks). A **Stop** hook does not emit
that; it emits the Stop decision (`{}` to allow the stop, or
`{"decision":"block","reason":…}` to bounce a message). So the generic check
rejected a perfectly correct Stop hook and marked the whole Claude deploy
failed.

## The fix

A dedicated Stop-contract self-check validates the Stop shape instead: the
output must be a JSON object; a `block` decision must carry a non-empty
`reason`. The generic `additionalContext` check is no longer wired to the Stop
hook. The regression test now asserts the deployed Stop script passes its own
self-check (the previous test's assertion was a no-op that always passed,
which is how this shipped).

No other client was affected; the observer, guard, recall, and primer hooks
already validated correctly.

## Known, next

The confirmation dialog on destructive actions (delete workspace, Delete all)
uses the browser `window.confirm()`, which does not render in the desktop
webview — a fix moving these to native dialogs is the next patch. Until then,
treat delete controls as immediate.
