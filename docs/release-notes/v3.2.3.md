# jawata-studio v3.2.3 — destructive actions actually prompt now (finding #8)

A dogfood patch, and an important one: it makes the confirmation dialogs on
destructive actions actually appear.

## The silent delete (finding #8)

Every destructive control — delete workspace, Delete all, the memory
clean/wipe actions, the runtime clean actions, quit-with-running-services —
guarded itself with the browser `window.confirm()`. In the desktop
(WebKitGTK/Tauri) webview that call **does not render a dialog**: it returns
without showing anything, so a "confirmed" action ran immediately with no
prompt and no chance to abort. A workspace was deleted this way with no
warning.

## The fix

All destructive confirmations now use the native dialog from
`tauri-plugin-dialog` (the same plugin the app already uses for file
open/save), through one `confirmDestructive()` helper. The dialog renders, and
Cancel aborts.

While fixing it, the two delete-workspace paths were de-duplicated: they used
to raise two confirmations (the list component and the app both prompted).
There is now exactly one native dialog per destructive action, and the
workspace-delete prompt states plainly that **your project source files on
disk are not deleted** — only the workspace's service, registration, and its
analysis-index directory.

No engine (jawata-mcp) change.
