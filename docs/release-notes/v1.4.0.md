# goja-studio v1.4.0 — Try-first enforcement (Sprint 22, PRE layer · hook half)

The status quo is **advertising**: goja's tools are visible, and a fluent agent ignores
them for `grep` + a hand-edit anyway (fluency over grounding). Advertising fails the way
"don't touch the credential file" fails — mostly obeyed, quietly slipped. An order needs a
**hook**, not a louder ad. v1.4.0 is the PRE-command *hook half*: guide the agent to the
right goja tool, then make it **try goja first — or say why goja can't do this**.

Ships together with **goja-mcp v1.6.0** (the PRE *tool half* — the same guide at the MCP
protocol level, plus the `analyze(kind=symbol)` workflow-collapse).

## The trigger→tool table (permanent prompt)

The deploy-written rule block now carries an intent→tool map — a "spam filter" the agent
reads before acting: search / usages / type / rename / move / extract / duplicate → the
exact goja tool or chain, and the **try-first, or justify** contract. It names the edit-side
mappings explicitly: `rename_symbol`, `move` / `move_in_hierarchy`, `extract`,
`generate(kind=copy_class)` + `extract(kind=superclass)`, `refactoring(action=plan)`.

## The PreToolUse guard — try-first, escapable, health-gated

The matcher widens to `Bash|Grep|Edit|Write|MultiEdit|mcp__goja.*`, and per-session state
lives at `~/.claude/goja-studio/trygate/<session_id>`:

- **goja-call logging** — every `mcp__goja*` call records its target tokens to the session
  file, so the guard knows what you've already looked up this session.
- **try-first search gate** — a `grep`/`rg` over a `.java` file is blocked *unless* you've
  already tried goja for that target this session, or you declare a fallback. Not blocked:
  text search over non-Java content (build files, configs, logs).
- **edit enforcement** — a hand `Edit`/`Write`/`MultiEdit` on an existing `.java` file is
  blocked with a redirect to the matching refactor tool. A brand-new file, a non-`.java`
  file, and (health-gated) **goja being down** all pass.
- **the escape valve** — every block is escapable with `goja-fallback: <why goja is
  inadequate here>`. Never stuck; just inconvenient to bypass without a reason.

## The versioned decision log — the scoring substrate

Each declared fallback is appended to `~/.claude/goja-studio/fallback.log` **stamped with
the goja engine version**. That version matters: "goja vX can't do Y" is a vX+1 feature
request, and when goja improves the category flips. The log is both a goja backlog and the
data a later scoring/classifier policy consumes — you cannot score goja-vs-fallback without
first *trying* goja, so enforcement is also data generation.

## Compatibility

Enforcement is health-gated (goja down → allow) and always escapable. Non-Java work is
untouched. No change to how any goja tool behaves.
