# goja-studio v2.0.2 — push-hook emits valid JSON (deployed-loop dogfood finding)

A **full deployed-loop dogfood** (the real hooks → the live v2.0 resident → the emitted
`hookSpecificOutput`) caught what neither the unit tests nor the direct-transport dogfood could:
the primer/recall `printf` used a bare `\n` header separator in its **format string**, which
`printf` turns into a **real newline** inside the `additionalContext` string value — producing
**invalid JSON**, which the client rejects (the injection silently fails).

## The fix

Both push scripts now use `\\n` in the printf format, so printf emits a literal `\n` **escape** —
valid inside a JSON string. Verified end-to-end: the corrected primer hook, run against the live
resident, emits JSON that parses cleanly (`additionalContext = "GOJA domain primer …:\n[domain_fact] …"`).

A regression test (`push_scripts_emit_escaped_newline_for_valid_json`) guards both templates.
`cargo test`: green.

## Why the earlier dogfood missed it

v2.0.1's dogfood drove the resident directly and inspected the *resident's* response. This one ran
the *deployed hook scripts* and validated *their emitted JSON* — the last mile. The lesson, twice
over: test the actual artifact over the actual wire. No mcp change — goja-mcp v2.0.0 is unaffected.
