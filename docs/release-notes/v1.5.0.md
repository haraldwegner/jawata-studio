# goja-studio v1.5.0 — PostToolUse observer (Sprint 22, POST layer · hook half)

The hook half of the POST layer. Where v1.4.0 (PRE) *enforces* try-first, v1.5.0 (POST) makes the
taken path pay off and **learns** from it: deploy now writes a managed **PostToolUse observer**
alongside the PreToolUse guard (undeploy strips both). It never blocks — PostToolUse can't — it
captures. Ships paired with **goja-mcp v1.7.0** (result steering + error hints).

## The versioned outcomes log — the scoring substrate

The observer appends three signals to `~/.claude/goja-studio/outcomes.log`, each stamped with the
GOJA engine version:

- **slip** — a declared `goja-fallback:` the PRE guard allowed → logged, plus a steering note
  injected into context ("verify next; a declared fallback is a GOJA feature request — if a newer
  GOJA can do it, prefer it").
- **read-ungrounded** — a `Read` of a `.java` file **not** preceded by a GOJA lookup this session
  (checked against the PRE guard's per-session state). Reading is too often legitimate to block, so
  this is **capture-only** — the utilization signal the search + edit gates can't see.
- **verify** — a compile / diagnostics / test event, logged to correlate with a preceding change.

Together with the PRE layer's versioned fallback log, these are the substrate a later classifier
(Sprint 26) and outcome-scoring pass (v2.1) consume — and the input for re-evaluating GOJA's own
gaps as it improves (a "goja vX can't do Y" entry retires when vX+1 can).

## Compatibility

Reactive, never blocks; preserves user hooks; idempotent + byte-stable re-deploy. Non-Java work is
left untouched.
