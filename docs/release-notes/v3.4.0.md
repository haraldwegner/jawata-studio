# jawata-studio v3.4.0 — launching the resident that retrieves by meaning

This release carries jawata-mcp **v3.4.0**, in which the experience store starts
answering questions phrased the way you asked them rather than the way the
knowledge was written down. The studio's own changes are small and both serve
that: the resident is launched so the embedder can run at full speed, and a
seat's recall now identifies itself so the quality counters can tell it apart.

## What changed here

**The resident is launched with the Vector API module** when the JVM has it.
jawata's embedder multiplies matrices for every cue, and `jdk.incubator.vector`
makes that roughly three times faster. Both paths produce identical vectors, so
this buys speed and risks nothing in results.

The flag is added only after probing the JVM that will run the resident,
because a JVM given `--add-modules` for a module it does not have **refuses to
start**. Passing it blindly would not have made residents slower, it would have
stopped them — on any JDK without the module, including the JDK where the
Vector API eventually leaves incubator and the module ceases to exist by that
name. Ask jawata's `health_check` which backend actually loaded; it reports the
one that ran, not the one that was requested.

**A driven seat's recall names itself.** A seat run and an ordinary prompt hook
issue the same call, so the resident cannot tell them apart. The runner now says
which it is, so the new quality counters attribute seat recalls correctly
instead of leaving that surface reading as "seats never recall".

## Upgrading

Nothing to do. Restart the residents from the studio after updating and they
pick up the new launch arguments.
